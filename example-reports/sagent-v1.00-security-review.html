Executive Summary

Sagent is a self-hosted, single-operator Telegram front-end for Claude Code: a Python supervisor (FastMCP/Starlette on :8081), a Python runner (Claude Agent SDK on :8082), a SvelteKit operator WebUI (:8086 host default), optional n8n/graphiti MCP integrations, Squid egress allowlisting, and a 13-service Docker Compose stack with deliberate east-west network segmentation. The project explicitly models one trusted operator — not multi-tenant adversarial isolation.

The codebase demonstrates strong security engineering for its threat model: OAuth credentials isolated on the supervisor with a stub decoy on the runner, default-deny tool permissions with Telegram per-call approval, env scrub allowlists, unified secret redaction, prompt-guard Unicode smuggling defense, hard Anthropic spend caps, constant-time secret comparison, and container capability baselines (cap_drop: ALL, read-only rootfs on most services). Residual risk concentrates in intentional network-trust boundaries (auth-exempt Anthropic proxy reachable from core/external peers) and deployment misconfiguration (WebUI published on all host interfaces without TLS, unauthenticated internal status endpoint).

Recommendation: Approve with fixes — suitable for intended single-operator self-hosting; do not expose WebUI or weaken network segmentation without mandatory hardening.

No Critical findings in the default compose path. High findings are exploitable when a trusted-tier container (n8n, graphiti-mcp) is compromised or when the WebUI is reachable from untrusted networks without TLS. Bind WebUI to loopback, enable TLS + secure cookies for LAN access, and treat n8n workflows as tier-1 trusted code.

App-specific review guidelines

  1. Every HTTP handler outside RPC middleware exemptions must enforce the correct secret (X-Sagent-RPC, X-Webui-Token, or Grafana Bearer) at the handler layer (src/telegram_mcp_server.py:173-221).
  2. Compromise of any container on external network is equivalent to Anthropic API access at operator subscription cost — treat n8n and graphiti-mcp as tier-1 trusted (docker-compose.yml:285-286).
  3. Telegram ingress must restrict to ALLOWED_USER_IDS — never empty in production compose.
  4. WebUI host port mapping should bind 127.0.0.1 unless TLS + WEBUI_SECURE_COOKIES=1 are configured (docker-compose.yml:623-625).
  5. All tool calls touching credentials, /proc/*/environ, or /run/secrets/** must hit SUPERVISOR_FORCE_PROMPT_RULES (src/permissions.py:58-84).
  6. Runner outbound must stay on Squid allowlist — never set SAGENT_DEV_PROXY_ALLOW_ALL in production.
  7. New /v1/webui/* mutation routes must call _require_webui_auth — middleware does not gate the mount (src/webui_routes.py:220-244).

[AUTH-001] Anthropic proxy exempt from RPC middleware

High
Category
Authentication / Trust boundary
Location
src/telegram_mcp_server.py:209-221
CWE
CWE-306, CWE-862

Description. The RPC auth middleware exempts all paths under /v1/anthropic/* because the Claude CLI subprocess cannot inject X-Sagent-RPC. Access control relies entirely on Docker network segmentation: any container attached to core or external can call the proxy. The proxy substitutes the operator's real OAuth bearer server-side (src/anthropic_proxy.py), billing the operator's Anthropic subscription.

Evidence.

exempt = (
    path == "/v1/health"
    or path == "/v1/anthropic"
    or path.startswith("/v1/anthropic/")
    or path == "/v1/webui/status"
    or path.startswith("/v1/webui/")
    ...
)
# Middleware comment (lines 179-184):
# Why the proxy is exempt: ... the CLI cannot inject the
# sagent-internal X-Sagent-RPC header. The network boundary is the access control.

Exploit scenario.

  1. Attacker compromises n8n (on external network per docker-compose.yml:294-296) via a malicious workflow import or supply-chain package.
  2. Workflow HTTP node sends POST http://supervisor:8081/v1/anthropic/v1/messages with arbitrary model/prompt — no RPC secret required.
  3. Proxy forwards with operator OAuth; spend meter accrues cost until cap triggers — but cap is tunable and may be set high.

Remediation. Add per-source authentication (e.g. mTLS or source-specific HMAC headers for n8n/graphiti); restrict external network membership; network-policy deny from n8n to /v1/anthropic/* if workflows do not require direct API access; document n8n as tier-1 trusted code with workflow signing review.

Verification. From n8n container, confirm Anthropic proxy calls are blocked or require additional auth; tests/test_anthropic_proxy.py continues to pass for runner path.

[SSRF-001] Auth-exempt Anthropic proxy enables paid-API abuse from internal peers

High
Category
SSRF / API abuse
Location
src/anthropic_proxy.py; docker-compose.yml:285-286
CWE
CWE-918

Description. Same root cause as AUTH-001, classified separately as SSRF-to-paid-API: internal containers can force the supervisor to make outbound Anthropic API calls on their behalf. This is not classic URL-fetch SSRF (no arbitrary URL parameter) but functionally equivalent for cost/abuse and data exfil via model prompts.

Evidence.

# docker-compose.yml:285-286
# n8n can call auth-exempt /v1/anthropic/* via the external network —
# use only trusted workflows (ADR 0012 → 0023; n8n is now isolated from the runner).

Exploit scenario. Compromised graphiti-mcp container (also on external) loops high-token requests through the proxy, exhausting operator quota or triggering spend-cap session cycling as a DoS side effect.

Remediation. Per-source rate limits on proxy (X-Sagent-Source tagging if present); separate OAuth scopes or API keys per integration tier; fail-closed network ACLs.

Verification. Integration test from non-runner external container returns 401/403 after fix; spend meter logs attribute requests by source.

[IAC-001] WebUI host port not loopback-scoped by default

High
Category
Infrastructure / Misconfiguration
Location
docker-compose.yml:623-625; webui/server.js:16-40
CWE
CWE-16, CWE-319

Description. Unlike n8n-gw (127.0.0.1:5678:5678) and grafana-gw, the WebUI publishes "${WEBUI_PORT:-8086}:3000" on all host interfaces. The container binds 0.0.0.0 by default. Plain HTTP is allowed when WEBUI_ALLOW_PLAIN_HTTP=1 (documented in compose comments). Session cookies use secure: false unless WEBUI_SECURE_COOKIES=1.

Evidence.

# docker-compose.yml
ports:
  - "${WEBUI_PORT:-8086}:3000"
  - "${WEBUI_HTTPS_HOST_PORT:-8443}:3443"

# webui/server.js — container listens on 0.0.0.0; startup guard only
# blocks non-loopback plain HTTP unless WEBUI_ALLOW_PLAIN_HTTP=1

Exploit scenario.

  1. Operator deploys on a VPS with default compose; WebUI listens on 0.0.0.0:8086 reachable from the public internet if security group allows.
  2. Attacker on same LAN sniffs HTTP session cookie (no Secure flag with default TLS off).
  3. Attacker uses stolen session to access operator dashboard, toggle MCP servers, approve pending requests if RPC secret configured, read FalkorDB graph summaries.

Remediation. Change default to "127.0.0.1:${WEBUI_PORT:-8086}:3000"; document Tailscale/SSH tunnel for remote access; set WEBUI_TLS=1 and WEBUI_SECURE_COOKIES=1 as production defaults in .env.example.

Verification. docker compose config shows loopback bind; external port scan shows closed unless explicitly configured; cookies have Secure flag when TLS enabled.

[AUTH-002] Unauthenticated GET /v1/webui/status

Medium
Location
src/webui_routes.py:286-365, 561
CWE
CWE-200

Description. Returns version, supervisor uptime, service probe states, credential presence flags (telegram, OpenAI, Claude OAuth, RPC secret), MCP server names/types/URLs, and approval risk rules — without X-Webui-Token. Intentional per ADR 0021 for dashboard polling, but any core network peer can enumerate operational metadata.

keys: dict[str, dict[str, Any]] = {
    "telegram_bot_token": {"present": bool(config.bot_token), ...},
    "openai_api_key": {"present": bool(config.openai_api_key), ...},
    ...
}
mcp_servers[name] = {"type": ..., "enabled": ..., "url": entry.get("url")}

Fix: Require X-Webui-Token for all /v1/webui/* including GET status; or restrict supervisor :8081 to webui-only network attachment; or serve status via unix socket.

[INJ-003] Telegram reply tool reads absolute paths

Medium

Location: src/telegram_mcp_server.py:332-364

Agent-supplied file paths in reply(files=[...]) are opened as absolute paths on the supervisor or fetched from runner. Sensitive paths are blocked by SUPERVISOR_FORCE_PROMPT_RULES (src/permissions.py:58-84), but mis-approval or rule gaps could exfil supervisor-local files to Telegram.

Fix: Restrict reply paths to runner workspace prefix; validate Path.resolve() stays under allowed roots before send.

[SEC-001] CTG_SKIP_ENV_SCRUB bypass

Medium

Location: src/env_scrub.py:77-100

if env.get(_SKIP_VAR):
    return []  # CTG_SKIP_ENV_SCRUB=1 bypasses scrub entirely

Operator debug hatch bypasses env scrub allowlist. If set in production compose, subprocesses inherit full environment including secrets popped elsewhere.

Fix: Fail boot if CTG_SKIP_ENV_SCRUB is set outside dev profile; log loud warning at minimum.

[SSRF-002] Dev egress wildcard bypass

Medium

Location: src/egress_broker.py, ADR 0033

SAGENT_DEV_PROXY_ALLOW_ALL inserts wildcard domain regex into Squid allowlist. In-process approval broker still gates new domains, but network-layer allowlist is weakened. Must never be enabled in production.

Fix: Compose profile gate (profiles: [dev]); refuse start if set without explicit dev flag.

[DEPS-001] Supply chain audit gap

Medium

Location: pyproject.toml, webui/package.json, pinned images in docker-compose.yml

Python deps include fastmcp==3.2.4, claude-agent-sdk>=0.1.61, httpx, pyyaml. Images pinned (sagent-supervisor:0.2.0, n8nio/n8n:1.119.1, caddy:2.11-alpine). No exhaustive CVE scan performed in this review; pre-commit secret scanner present but not dependency audit.

Fix: Add pip audit / npm audit / OSV-Scanner to CI; pin Python deps with hashes where feasible.
