Executive Summary

Taiga Backend is a Django 3.2 REST API for agile project management (stories, tasks, wiki, webhooks, VCS integrations). The codebase implements solid project-scoped RBAC, JWT authentication with token denylist, parameterized SQL for search, outbound webhook SSRF blocking by default, and markdown sanitization via bleach.

However, default configuration is unsafe for production: a hardcoded SECRET_KEY, wildcard ALLOWED_HOSTS, default database/broker credentials, disabled API throttling, Celery pickle serialization, and GitLab inbound webhooks that accept any source IP when GITLAB_VALID_ORIGIN_IPS is empty. Django 3.2 is end-of-life and no longer receives security patches.

Recommendation: Approve with fixes — do not deploy to production without mandatory hardening.

Block production deployment until Critical/High configuration and operational items are resolved. The application logic is generally sound; risk concentrates in deployment defaults and missing rate limits.

App-specific review guidelines

  1. Every /api/v1/* handler must enforce authentication before side effects (JWT or explicit AllowAny).
  2. Project ID in URLs must be authorized via HasProjectPerm or filter backends — never trust body alone.
  3. Inbound VCS hooks (github-hook, gitlab-hook, etc.) are unauthenticated; verify HMAC/IP allowlists.
  4. Outbound webhook URLs must pass validate_private_url() unless explicitly overridden.
  5. All .extra(where=...) and cursor.execute paths must use bound parameters.
  6. Production must override settings/common.py defaults — never ship with committed SECRET_KEY.

Critical & High Findings

[CFG-002] Hardcoded SECRET_KEY in default settings

Critical
Category
Secrets / Cryptographic failure
Location
settings/common.py:249
CWE
CWE-798, CWE-321

Description. The default Django SECRET_KEY is committed in source control. Anyone deploying without overriding this value shares the same signing key as every other default installation, enabling JWT/session forgery and cryptographic bypass.

Evidence.

SECRET_KEY = "aw3+t2r(8(0kkrhg8)gx6i96v5^kv%6cfep9wxfom0%7dy0m9e"

Exploit scenario.

  1. Attacker identifies a Taiga instance using default or leaked configuration.
  2. Attacker forges JWT tokens or session cookies signed with the known key.
  3. Attacker gains authenticated access as any user, including administrators.

Remediation. Generate a unique SECRET_KEY per environment via environment variable (docker/config.py uses TAIGA_SECRET_KEY). Fail startup if unset. Never commit production keys. Rotate keys on compromise.

Verification. Confirm settings.SECRET_KEY is not the default value in production; add CI lint rejecting known default key string.

[CFG-001] ALLOWED_HOSTS accepts any host

High
Category
Security misconfiguration
Location
settings/common.py:17
CWE
CWE-16

Description. ALLOWED_HOSTS = ["*"] disables Django's Host header validation, enabling host-header poisoning and cache-poisoning attacks when deployed behind misconfigured proxies.

ALLOWED_HOSTS = ["*"]

Exploit scenario. Attacker sends requests with a malicious Host header; password-reset or absolute URLs in emails may point to attacker-controlled domains.

Remediation. Set explicit hostnames in production config (see settings/config.py.prod.example).

[CFG-003] Default database and message broker credentials

High
Category
Secrets
Location

DATABASES = { "default": { "NAME": "taiga", "USER": "taiga", "PASSWORD": "taiga", ... } }
CELERY_BROKER_URL = 'amqp://guest:guest@localhost:5672//'

Exploit scenario. Attacker with network access to PostgreSQL or RabbitMQ on a misconfigured deployment gains full data access or can inject Celery tasks.

Remediation. Use strong unique credentials; restrict network access; enable TLS for broker connections in production.

[AUTH-001] Login and recovery brute force — throttling disabled

High
Category
Authentication
Location
settings/common.py:485-496
"login-fail": None,
"register-success": None,
...

Description. Throttle infrastructure exists (taiga/base/throttling.py, LoginFailRateThrottle) but default rates are None, which causes CommonThrottle to allow unlimited requests.

Exploit scenario. Attacker performs credential stuffing against /api/v1/auth and password recovery spam without rate limits.

Remediation. Set production rates, e.g. "login-fail": "5/minute", "register-success": "3/hour", "anon-write": "60/minute".

[RATE-001] Global API throttling disabled

High
Category
API security
Location
settings/common.py:485-496

All scopes except import-dump-mode are None. Enables API abuse, resource exhaustion, and registration spam on public instances.

Remediation. Configure per-scope limits appropriate to deployment size; consider reverse-proxy rate limiting (nginx, Cloudflare).

[AUTHZ-001] GitLab inbound webhooks accept any source IP

High
Category
Authorization / Access control
Location
settings/common.py:597; taiga/hooks/gitlab/api.py:47-61
GITLAB_VALID_ORIGIN_IPS = []

# In gitlab/api.py — when valid_origin_ips is empty, matching_origin_ip stays True
if valid_origin_ips:
    matching_origin_ip = len(all_matching_cidrs(origin_ip, valid_origin_ips)) > 0

Exploit scenario. Attacker who obtains the GitLab hook URL and project secret (?project=<id>&key=<secret>) can POST from any IP and trigger issue/task mutations — IP allowlist is not enforced by default.

Remediation. Populate GITLAB_VALID_ORIGIN_IPS with GitLab instance CIDRs, or enforce IP filtering at the reverse proxy. Compare with Bitbucket's default CIDR allowlist in the same settings file.

[DESER-001] Celery pickle serialization

High
Category

Location
settings/common.py:53-55
CWE
CWE-502
CELERY_ACCEPT_CONTENT = ['pickle', ]
CELERY_TASK_SERIALIZER = "pickle"
CELERY_RESULT_SERIALIZER = "pickle"

Exploit scenario. Compromised RabbitMQ or message injection leads to arbitrary code execution on Celery workers when pickle is deserialized.

Remediation. Switch to JSON: CELERY_TASK_SERIALIZER = "json", CELERY_ACCEPT_CONTENT = ["json"]. Audit task signatures for pickle-only types.

[AUTH-004] Password recovery user enumeration

Medium

Location: taiga/users/api.py:157-172

Invalid username returns error; valid user returns {"detail": "Mail sent successfully!"}. Attacker enumerates registered emails.

Fix: Always return identical success response; send email only when user exists.

[AUTH-005] Recovery token has no expiry

Medium

[SSRF-002] Private-address webhooks admin toggle

Medium

Location: settings/common.py:602, taiga/webhooks/tasks.py:76-78

Default WEBHOOKS_ALLOW_PRIVATE_ADDRESS = False is safe. When enabled via docker/config.py env, project admins can target internal metadata endpoints (e.g. cloud IAM).

[XSS-002] |safe filter in email subject templates

Medium

Location: taiga/projects/notifications/templates/emails/**/**-subject.jinja

User-controlled project names and story subjects bypass escaping in email subjects — phishing/XSS in email clients.

Fix: Remove |safe or HTML-escape user content in notification templates.

Verified Safe Controls

  • JWT validation + denylist — taiga/auth/authentication.py, token denylist module; HS256, active user check
  • Project RBAC — HasProjectPerm, membership roles, anon/public permission arrays
  • Queryset isolation — PermissionBasedFilterBackend scopes lists to visible projects
  • Outbound webhook SSRF block — validate_private_url() in taiga/webhooks/tasks.py:76-78
  • GitHub inbound HMAC — hmac.compare_digest with project secret
  • Bitbucket IP allowlist — Default Atlassian CIDR ranges configured
  • SQL parameterization — Search/filter uses bound parameters via to_tsquery()
  • Markdown sanitization — bleach.clean() in taiga/mdrender/service.py
  • Public registration disabled — PUBLIC_REGISTER_ENABLED = False
  • Password hashing — PBKDF2 via Django defaults

Recommended Next Steps

Production blockers (mandatory)

  1. Override SECRET_KEY, ALLOWED_HOSTS, DB and RabbitMQ credentials (CFG-001/002/003)
  2. Enable throttle rates: login-fail, register-success, anon-write, user-write (AUTH-001, RATE-001)
  3. Configure GITLAB_VALID_ORIGIN_IPS or proxy-level IP filtering (AUTHZ-001)
  4. Switch Celery to JSON serialization (DESER-001)
  5. Plan Django 4.2+ upgrade with regression testing (DEP-001)

Recommended (non-blocking)

  1. Add password recovery token TTL (AUTH-005)
  2. Uniform password recovery responses (AUTH-004)
  3. Restrict attachment MIME types (FILE-002)
  4. Remove |safe from email subject templates (XSS-002)
  5. Fix CORS to explicit origin allowlist (CFG-004)
  6. Add CI gate: reject default SECRET_KEY string in deployed configs
