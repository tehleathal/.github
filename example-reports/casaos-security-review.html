Executive Summary

CasaOS is an open-source personal cloud / home-server OS written in Go (Echo HTTP framework). The main service binds to 127.0.0.1 and registers routes with an external CasaOS Gateway that proxies public traffic. The architecture exposes filesystem management, Samba shares, cloud storage (rclone), ZeroTier networking, system control (update/shutdown), and WebSocket SSH — all intended for a single home administrator.

The review identified four Critical issues that chain into full host compromise: JWT authentication is skipped for all requests proxied through the Gateway (localhost Skipper), file APIs read/write arbitrary paths on the host without a jail, and system updates execute curl | bash from configurable URLs. Additional High findings include unauthenticated debug endpoints, SSRF proxies, Samba shares with anonymous guest write mapped to root, and hardcoded signing secrets.

Recommendation: Block — do not expose CasaOS Gateway to untrusted networks until Critical findings are remediated.

The AUTH-001 localhost JWT bypass effectively nullifies authentication for all Gateway-proxied API traffic. Combined with PATH-001/002, an unauthenticated attacker on the LAN (or internet if the Gateway is reachable) can read /etc/shadow, write cron jobs, trigger remote code execution via POST /v1/sys/update, and control ZeroTier/rclone integrations.

App-specific review guidelines

  1. JWT middleware must not skip validation for Gateway-proxied requests (route/v1.go:45-47).
  2. All file operations must use JoinBasePath / IsSubPath against an allowlisted base (e.g. /DATA).
  3. System updates must use signed packages — never curl | bash (service/system.go:382-385).
  4. SSRF endpoints (/v1/sys/proxy, /v1/other/search, /v1/zt/*) must block private/loopback IPs.
  5. Samba shares must not default to anonymous guest write with force user = root.
  6. Debug and version endpoints must require authentication in production builds.

Critical & High Findings

[AUTH-001] JWT authentication bypass via localhost Skipper

Critical
Category
Authentication
Location
route/v1.go:44-47; route/v2.go:71-74; main.go:129,152-156
CWE
CWE-306, CWE-287

Description. JWT middleware skips validation when RealIP() is 127.0.0.1 or ::1. The main service listens only on localhost and the Gateway proxies all public traffic to that address, so every external request appears as localhost and bypasses JWT entirely.

Evidence.

v1Group.Use(echo_middleware.JWTWithConfig(echo_middleware.JWTConfig{
    Skipper: func(c echo.Context) bool {
        return c.RealIP() == "::1" || c.RealIP() == "127.0.0.1"
    },
    ...
}))
// main.go — Gateway registers routes to localhost listener
listener, err := net.Listen("tcp", net.JoinHostPort(LOCALHOST, "0"))
err = service.MyService.Gateway().CreateRoute(&model.Route{
    Path: apiPath, Target: "http://" + listener.Addr().String(),
})

Exploit scenario.

  1. Attacker reaches CasaOS Gateway on LAN or internet (default home-server deployment).
  2. Gateway forwards request to Main; Echo sees RealIP() == 127.0.0.1.
  3. JWT Skipper returns true — no token required for /v1/file/*, /v1/sys/update, /v1/sys/stop, etc.

Remediation. Remove the localhost Skipper. Authenticate at the Gateway layer and pass verified identity via non-spoofable headers, or validate JWT on every request regardless of source IP.

Verification. Send GET /v1/file/content?path=/etc/hostname through Gateway without Authorization header — must return 401.

[PATH-001] Arbitrary file read — no path jail

Critical
Category
Broken access control
Location
route/v1/file.go:87-116,225-260; route/v2.go:166-169
CWE
CWE-22, CWE-200

Description. File content and download handlers pass user-supplied path query parameters directly to ioutil.ReadFile, os.Open, and http.ServeFile without restricting to an allowlisted base directory.

Evidence.

func GetFilerContent(ctx echo.Context) error {
    filePath := ctx.QueryParam("path")
    ...
    info, err := ioutil.ReadFile(filePath)
}
// route/v2.go InitFile handler
filePath := r.URL.Query().Get("path")
http.ServeFile(w, r, filePath)

Exploit scenario. With AUTH-001 bypass: GET /v1/file/content?path=/etc/shadow returns password hashes. Also readable: /etc/casaos/casaos.conf, SSH keys, rclone cloud tokens.

Remediation. Enforce base-path containment using JoinBasePath("/DATA", userPath) and post-join prefix validation via IsSubPath.

Verification. Request path=../../../etc/passwd — must return 403 regardless of auth state.

[INJ-002] Remote code execution via curl | bash system update

Critical
Category
Injection / RCE
Location
service/system.go:373-386; route/v1/system.go:74
CWE
CWE-94, CWE-829

Description. System update executes remote shell scripts piped from curl without signature verification. Update URL is configurable via casaos.conf.

Evidence.

func (s *systemService) UpdateSystemVersion(version string) {
    if len(config.ServerInfo.UpdateUrl) > 0 {
        go command.OnlyExec("curl -fsSL " + config.ServerInfo.UpdateUrl + " | bash")
    } else {
        go command.OnlyExec("curl -fsSL https://get.casaos.io/update?t=" + osRelease["MANUFACTURER"] + " | bash")
    }
}
// route/v1.go — POST /v1/sys/update triggers this

Exploit scenario. With AUTH-001: POST /v1/sys/update runs arbitrary remote code as the CasaOS service user (often root). Attacker who can modify UpdateUrl in config achieves persistent RCE on next update trigger.

Remediation. Use signed packages (deb/rpm) with GPG verification; never pipe curl output to bash.

Verification. Update flow uses package manager only; network MITM cannot execute arbitrary shell.

[DEFAULT-001] Samba shares with anonymous guest write as root

High
Category
Misconfiguration
Location
route/v1/samba.go:86-92; service/shares.go:77-91
CWE
CWE-276, CWE-732

Description. New Samba shares default to anonymous access with world-writable permissions and force user = root in generated smb.conf.

shareDBModel.Anonymous = true
os.Chmod(v.Path, 0o777)
// service/shares.go generated config:

Exploit scenario. Any LAN client writes to shared folders as root-mapped Samba user — data destruction or planting malicious files.

Medium Findings

[AUTH-003] Multiple unauthenticated API endpoints

High

Location: route/v1.go:34-41

Public routes: /ping, /v1/sys/version/check, /v1/sys/version/current, /v1/recover/:type (OAuth callback). Version endpoints aid fingerprinting; OAuth recover may be CSRF-vulnerable without session-bound state.

Fix: Restrict version endpoints; bind OAuth state to authenticated session.

[PATH-003] Path sanitization utilities exist but are unused

High

Location: pkg/utils/path.go:76-80

JoinBasePath and IsSubPath reject ../ traversal but are never called from route handlers — developers assumed safety that does not exist.

Fix: Apply consistently in all file endpoints.

[SSRF-002] Search agent fetches user-supplied URL

High

Location: route/v1/other.go:12-21; service/other.go:103-111

POST /v1/other/search with {"url":"http://127.0.0.1:6379/"} probes internal services via resty client.

Fix: Same SSRF controls as SSRF-001; restrict to known search provider patterns.

[SSRF-003] ZeroTier API raw proxy

High

Location: route/v1/zerotier.go:20-67

Proxies arbitrary paths to local ZeroTier API using /var/lib/zerotier-one/authtoken.secret. Attacker can join networks or modify controller config.

Fix: Replace with narrow validated wrapper endpoints; do not expose raw ZT API.

[INJ-001] Command injection via network interface name

High

Location: service/system.go:264

command.OnlyExec("... CatNetCardState " + name) concatenates interface name into shell script invocation without quoting.

Fix: Validate name against ^[a-zA-Z0-9._-]+$; pass as quoted argument.

[SEC-001] Hardcoded HMAC signing key

High

Location: internal/sign/sign.go:35

instance = sign.NewHMACSign([]byte("token"))

Static key enables forging signed URLs if signing protects file/share access.

Fix: Load per-installation key from secure store; rotate on install.

[DEFAULT-003] SSH credentials in URL query parameters

High

Location: route/v1/ssh.go:53-55

GET /v1/sys/wsssh?username=&password=&port= — credentials logged in access logs and browser history.

Fix: Accept credentials via POST body over TLS; prefer SSH keys.

[AUTH-004] JWT token accepted in query string

Medium

Location: route/v1.go:58-64; route/v2.go file handler

Tokens leak via Referer, logs, and browser history. Combined with PATH-001 enables file download via leaked token.

Fix: Accept tokens only via Authorization: Bearer header.

[CSRF-001] No CSRF protection; permissive CORS

Medium

Location: route/v1.go:20-26

AllowOrigins: ["*"] with AllowCredentials: true — invalid CORS combination. No CSRF tokens on state-changing endpoints.

Fix: Explicit origin allowlist; CSRF tokens for browser flows; OAuth state binding.

[SEC-002] Hardcoded Dropbox OAuth credentials

Medium

Location: drivers/dropbox/meta.go:12-13

Default app key/secret shared across all CasaOS installations.

Fix: Per-installation OAuth client registration.

[DEFAULT-002] WebSocket origin validation disabled

Medium

Location: route/v1/ssh.go:23-27; route/v1/file.go:69-73

CheckOrigin: func(r *http.Request) bool { return true } — cross-site WebSocket hijacking possible.

Fix: Validate Origin against CasaOS UI allowlist.

[DOCKER-001] Unrestricted rclone unix socket access

Medium

Location: pkg/utils/httper/drive.go:47-86

Cloud mount endpoints invoke rclone RC API via /var/run/rclone/rclone.sock. With AUTH-001, attacker mounts attacker-controlled remotes or reads stored tokens.

Fix: Restrict socket permissions; validate mount paths; require strong auth.

[DEFAULT-004] Startup script execution from config directory

Medium

Location: main.go:201-202

command.ExecuteScripts(/etc/casaos/start.d) — writable via PATH-002 yields persistent code execution on restart.

Fix: Restrict directory permissions; sign scripts or use systemd with integrity checks.

Verified Safe Controls

  • Main service binds localhost only — main.go:129 uses LOCALHOST = "127.0.0.1" (defense in depth if Gateway is compromised)
  • HTTP slowloris mitigation — ReadHeaderTimeout: 5s on server
  • ECDSA JWT validation (when not skipped) — jwt.Validate with public key from runtime path
  • Path traversal helpers available — JoinBasePath rejects ../ (needs adoption in handlers)
  • GORM parameterized DB queries — Where("id = ?", id) in connections service
  • System reboot/shutdown use fixed args — exec.Command("init", "6") without user input
  • Samba connection list omits passwords — selective field return in API responses

Recommended Next Steps

Deployment blockers (mandatory)

  1. Remove JWT localhost Skipper and verify auth end-to-end through Gateway (AUTH-001)
  2. Enforce filesystem jail on all file endpoints using existing JoinBasePath helpers (PATH-001, PATH-002, PATH-003)
  3. Replace curl | bash updates with signed packages (INJ-002)
  4. Disable or restrict SSRF endpoints and ZeroTier raw proxy (SSRF-001, SSRF-002, SSRF-003)
  5. Fix Samba defaults — no anonymous guest write with force user root (DEFAULT-001)

Recommended (non-blocking)

  1. Remove or protect /v1/sys/debug and public version endpoints (AUTH-002, AUTH-003)
  2. Remove query-string JWT acceptance (AUTH-004)
  3. Rotate hardcoded HMAC and OAuth secrets (SEC-001, SEC-002)
  4. Fix CORS configuration; add CSRF for browser flows (CSRF-001)
  5. Run govulncheck ./... in CI (A06)
  6. Re-run this review after auth and filesystem fixes land
